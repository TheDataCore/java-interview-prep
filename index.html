OOP — Object Oriented Programming
  • What are the 4 pillars of OOP? Give real project examples for each.
    Easy
    ▼
    1. Encapsulation — hiding internal state; expose only via controlled API.
    Example: BankAccount has private balance; only deposit() and withdraw() modify it, with validation logic inside.

    2. Inheritance — IS-A relationship. Reuse and extend behaviour via extends.
    Example: ElectricVehicle extends Vehicle — inherits startEngine() but overrides with startMotor().

    3. Polymorphism — same interface, different behaviour.
    Compile-time: method overloading (same name, different params).
    Runtime: method overriding via @Override — Shape[] shapes; shapes[0].area() calls Circle.area() or Square.area() based on actual type.

    4. Abstraction — expose WHAT, hide HOW. Via abstract classes and interfaces.
    Example: PaymentGateway interface with processPayment(); StripeGateway and PaypalGateway implement it differently. 
    // Polymorphism + Abstraction together
public interface PaymentGateway {
    PaymentResult processPayment(PaymentRequest req);
    boolean refund(String transactionId);
}

@Service("stripe")
public class StripeGateway implements PaymentGateway {
    @Override
    public PaymentResult processPayment(PaymentRequest req) { /* Stripe API */ }
    @Override
    public boolean refund(String txId) { /* Stripe refund */ }
}

  • What is method overloading vs overriding? Explain with covariant return types.
    Medium
    ▼
    Overloading (Compile-time polymorphism): same method name, different parameter list in the same class. Return type alone cannot distinguish.
    Overriding (Runtime polymorphism): subclass re-implements parent method with same signature. Annotate with @Override. Covariant return: subclass can return a more specific type.

class Printer {

    void print(int n) { }
    void print(String s, int copies) { }
}

class Animal { Animal create() { return new Animal(); } }
class Dog extends Animal {
    @Override
    Dog create() { return new Dog(); } // covariant — Dog is subtype of Animal
}

// Dynamic dispatch — resolved at runtime
Animal a = new Dog();
a.sound(); // calls Dog.sound(), not Animal.sound()

Collections Framework
  • Explain HashMap internal working — hashing, collision, treeification
    Hard
    ▼
    HashMap internally uses an array of Node<K,V>[] called table. Default capacity = 16, load factor = 0.75.

    put(key, value) flow:
    1. hashCode() called on key → hash spread with XOR shift: hash = (h = key.hashCode()) ^ (h >>> 16)
    2. Index = hash & (n-1) where n = table length (always power of 2 → fast modulo)
    3. If slot empty → insert Node. If occupied (collision) → traverse linked list, compare keys with equals()
    4. If chain length > 8 AND table size ≥ 64 → convert to Red-Black Tree (O(log n) vs O(n))
    5. If size > capacity × 0.75 → resize: double capacity, rehash all entries

    Why equals() AND hashCode() contract matters: Two equal objects must have same hashCode. If you override equals() without hashCode(), equal objects go to different buckets → broken Map behavior.
    // Correct custom key
public class OrderKey {
    private String orderId;
    private String userId;
    
    @Override
    public int hashCode() {
        return Objects.hash(orderId, userId);
    }
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof OrderKey ok)) return false;
        return orderId.equals(ok.orderId) && userId.equals(ok.userId);
    }
}

    Easy
    ▼
    OperationArrayListLinkedList
    get(i) / set(i)O(1) — backed by arrayO(n) — must traverse

  • Difference: HashSet, LinkedHashSet, TreeSet. How is HashSet backed internally?
    Medium
    ▼
    HashSet — backed by a HashMap (values are a dummy PRESENT object). No ordering. O(1) add/remove/contains.
    LinkedHashSet — backed by LinkedHashMap. Preserves insertion order. Slightly slower.
    TreeSet — backed by TreeMap (Red-Black Tree). Sorted order. O(log n). Implements NavigableSet — headSet(), tailSet(), floor(), ceiling().

    Since HashSet uses HashMap internally, the same hashCode()+equals() contract applies. All elements must correctly implement these for Set to work. 
    TreeSet<Integer> ts = new TreeSet<>();
ts.addAll(List.of(5,3,8,1,9));
System.out.println(ts.headSet(6)); // [1, 3, 5]
System.out.println(ts.floor(4));   // 3 (greatest ≤ 4)
System.out.println(ts.ceiling(4)); // 5 (smallest ≥ 4)
  • ConcurrentHashMap internals — how thread safety works in Java 8
    Hard
    ▼
    Java 7: ConcurrentHashMap used Segment[] locking (16 segments). Each segment was a mini HashMap with a lock.
    Java 8: Replaced with CAS (Compare-And-Swap) + synchronized on bucket head only. Much finer-grained.

    Java 8 Internals:
    • Reads: Completely lock-free (volatile Node array)
    • Empty bucket insert: CAS operation — atomic, no lock
    • Collision (non-empty bucket): synchronized on the head Node — only locks that one bucket
    • No null keys or values allowed (unlike HashMap)
    • computeIfAbsent(), merge(), compute() are atomic operations
    ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

// Atomic increment — use compute, not get+put
map.compute("count", (k, v) -> v == null ? 1 : v + 1);

// Or merge
map.merge("count", 1, Integer::sum);

// NOT thread-safe even with CHM:
int v = map.get("x"); map.put("x", v + 1); // race condition!

  • Streams — explain lazy evaluation, short-circuit, and parallel streams
    Hard
    ▼
    Lazy evaluation: Intermediate operations (filter, map, sorted) create a pipeline but do NOT execute until a terminal operation is called. This means zero work if there's no terminal op. 
    Stream<String> s = names.stream()
    .filter(n -> { System.out.println("filter: "+n); return n.length() > 3; })
    .map(String::toUpperCase);
// Nothing prints yet — no terminal op
s.findFirst(); // NOW it executes, and stops after first match (short-circuit)
    Short-circuit operations: findFirst(), findAny(), anyMatch(), allMatch(), noneMatch(), limit() — stop processing once result is determined.

    Parallel Streams: stream().parallel() or parallelStream() splits source using ForkJoinPool.commonPool(). Good for CPU-bound, independent operations on large datasets (>10K elements). Bad for: small datasets (overhead), IO-bound tasks, stateful operations, non-thread-safe collectors. 
    // Good use case
long count = largeList.parallelStream()
    .filter(item -> expensiveCheck(item))
    .count();

// Danger: shared mutable state with parallel
List<String> result = new ArrayList<>(); // NOT thread-safe!
names.parallelStream().forEach(result::add); // race condition!
// FIX: .collect(Collectors.toList()) instead

  • Thread lifecycle, ExecutorService, and thread pool sizing
    Hard
    ▼
    Thread states: NEW → RUNNABLE → BLOCKED/WAITING/TIMED_WAITING → TERMINATED

    ExecutorService is preferred over raw Thread — manages pool, reuse, lifecycle. 
    // Fixed pool — good for CPU-bound tasks
ExecutorService pool = Executors.newFixedThreadPool(
    Runtime.getRuntime().availableProcessors()
);

// Cached pool — good for short-lived IO tasks (creates threads on demand)
ExecutorService io = Executors.newCachedThreadPool();

// Scheduled
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);
scheduler.scheduleAtFixedRate(() -> cleanup(), 0, 1, TimeUnit.HOURS);

// Submit with Future
Future<String> future = pool.submit(() -> fetchData());
String result = future.get(5, TimeUnit.SECONDS); // blocking with timeout

// Always shutdown!
pool.shutdown();
pool.awaitTermination(60, TimeUnit.SECONDS);
    Thread pool sizing:
    CPU-bound: N+1 threads (N = CPU cores, +1 for page faults)
    IO-bound: N × (1 + wait_time/service_time) — can be much larger (e.g., 100+)
  • Deadlock — causes, detection, and prevention strategies
    Hard
    ▼
    Deadlock conditions (all 4 must hold): Mutual exclusion, Hold and wait, No preemption, Circular wait.
    Break any one condition to prevent deadlock. 
    // Classic deadlock
Object lockA = new Object(), lockB = new Object();
Thread t1 = new Thread(() -> {
    synchronized(lockA) {
        Thread.sleep(100);
        synchronized(lockB) { } // waits for lockB
    }
});
Thread t2 = new Thread(() -> {
    synchronized(lockB) {
        Thread.sleep(100);
        synchronized(lockA) { } // waits for lockA — DEADLOCK!
    }
});

// Prevention: always acquire locks in same order
Thread t1fixed = new Thread(() -> {
    synchronized(lockA) { synchronized(lockB) { } }
});
Thread t2fixed = new Thread(() -> {
    synchronized(lockA) { synchronized(lockB) { } } // same order
});

// Detection: jstack, thread dumps, or tryLock with timeout
ReentrantLock lock = new ReentrantLock();
if (lock.tryLock(2, TimeUnit.SECONDS)) {
    try { /* work */ } finally { lock.unlock(); }
} else { /* handle timeout — avoid deadlock */ }
  • JVM memory model — heap, stack, metaspace, GC types
    Hard
    ▼
    JVM Memory Areas:
    • Heap — objects, class instances. Divided into Young Gen (Eden + S0/S1) and Old Gen (Tenured)
    • Stack — per-thread; stores method frames, local variables, partial results
    • Metaspace (Java 8+, replaced PermGen) — class metadata, static fields
    • Code Cache — JIT-compiled native code
    • Direct Memory — off-heap, used by NIO ByteBuffer

    GC Types:
    • Serial GC — single thread, small heaps
    • Parallel GC — multi-thread STW (Stop-The-World), throughput focus
    • G1 GC — default since Java 9. Region-based, predictable pause times
    • ZGC/Shenandoah — concurrent, sub-millisecond pauses, Java 15+

    OOM types: java.lang.OutOfMemoryError: Java heap space (objects), Metaspace (class loading leak), GC overhead limit exceeded (GC taking >98% time)

Critical Annotations — Know Every Single One

@SpringBootApplication

= @Configuration + @EnableAutoConfiguration + @ComponentScan. Entry point. Scans from its package downward.

@RestController

= @Controller + @ResponseBody. All methods return JSON/XML. No ViewResolver involved.

@Service / @Repository / @Component

All create beans. @Repository adds exception translation (DB → DataAccessException). @Service is semantic. @Component is generic.

@Autowired

Injects by type. On constructor (preferred), setter, or field. Use @Qualifier("beanName") when multiple candidates exist.

@Transactional

Wraps in DB transaction. PROPAGATION_REQUIRED default (reuse existing). readOnly=true optimizes queries. rollbackFor = specific exceptions.

@ControllerAdvice / @ExceptionHandler

Global exception handling. @ControllerAdvice applies to all controllers. @ExceptionHandler maps exception type to handler method.

@Value / @ConfigurationProperties

@Value("${app.key}") injects single property. @ConfigurationProperties(prefix="app") maps group of props to POJO — preferred.

@Cacheable / @CacheEvict / @CachePut

@Cacheable — return cached value if exists. @CacheEvict — remove from cache. @CachePut — update cache. Uses SpEL for key.

@Scheduled

Schedule methods: cron, fixedRate, fixedDelay. Needs @EnableScheduling on config class.

@Async

Run method asynchronously in thread pool. Needs @EnableAsync. Returns void or Future/CompletableFuture.

@Profile

Activate beans for specific profiles (dev, prod, test). Set via spring.profiles.active property or env var.

@ConditionalOnProperty / @ConditionalOnClass

Auto-config conditions. Bean only created if property/class exists. Use for feature flags and conditional wiring.

  • Explain Spring IoC container — BeanFactory vs ApplicationContext, bean scopes
    Medium
    ▼
    IoC (Inversion of Control): The framework creates and manages objects (beans), not the application code. You declare what you need; Spring injects it.

    BeanFactory — basic IoC container. Lazy initialization (beans created on first request). Rarely used directly.
    ApplicationContext — extends BeanFactory. Adds: event publishing, i18n, resource loading, AOP, eager singleton initialization. Always use this in Spring Boot.

    Bean Scopes:
    • singleton (default) — one instance per ApplicationContext. Shared across the app.
    • prototype — new instance each time bean is requested. Not managed after creation (no destroy callback).
    • request — one per HTTP request (web apps only)
    • session — one per HTTP session
    • application — one per ServletContext

  • How does @Transactional work internally? What are propagation types?
    Hard
    ▼
    Spring creates a proxy (JDK dynamic proxy or CGLIB) around the bean. The proxy intercepts method calls and manages the transaction via PlatformTransactionManager.

    Important gotchas:
    1. @Transactional only works on public methods (proxy can't intercept private/package)
    2. Self-invocation bypasses the proxy: calling another @Transactional method within the same class won't start a new transaction
    3. Default rollback only on RuntimeException, not checked exceptions. Use rollbackFor=Exception.class for all.

    Propagation types:
    TypeBehaviour
    REQUIRED (default)Join existing TX or create new one
    REQUIRES_NEWAlways create new TX, suspend existing
    NESTEDSavepoint in existing TX — partial rollback possible
    SUPPORTSJoin if exists, proceed without TX if not
    NOT_SUPPORTEDSuspend TX if exists, run without
    MANDATORYMust have existing TX, else throw
    NEVERMust NOT have TX, else throw
    @Transactional(propagation = Propagation.REQUIRES_NEW, 
               rollbackFor = Exception.class,
               timeout = 30,
               readOnly = false)
public Order createOrder(CreateOrderRequest req) { ... }
  • What is AOP? Explain Aspect, Advice, Pointcut, JoinPoint with a real example
    Hard
    ▼
    AOP (Aspect Oriented Programming) separates cross-cutting concerns (logging, security, transactions, metrics) from business logic.

    • Aspect — the class containing cross-cutting logic (@Aspect)
    • JoinPoint — a point in execution (method call, exception thrown)
    • Pointcut — expression selecting which JoinPoints to intercept
    • Advice — code to run: @Before, @After, @AfterReturning, @AfterThrowing, @Around
    @Aspect
@Component
public class ExecutionTimeAspect {

    // Pointcut: any method in service package
    @Pointcut("execution(* com.myapp.service.*.*(..))")
    public void serviceLayer() {}

    @Around("serviceLayer()")
    public Object logExecutionTime(ProceedingJoinPoint jp) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            Object result = jp.proceed(); // execute actual method
            long elapsed = System.currentTimeMillis() - start;
            log.info("{} executed in {}ms", jp.getSignature().getName(), elapsed);
            return result;
        } catch (Exception e) {
            log.error("Exception in {}: {}", jp.getSignature(), e.getMessage());
            throw e;
        }
    }

    @Before("@annotation(com.myapp.annotation.RequiresAdmin)")
    public void checkAdminRole(JoinPoint jp) {
        if (!SecurityContext.isAdmin()) throw new AccessDeniedException("Admin required");
    }
}
    Spring's @Transactional itself is implemented with AOP — a Pointcut on @Transactional methods, Around advice managing the TX.

  • Spring Boot Security — complete JWT implementation walkthrough
    Hard
    ▼
    // JWT Utility
@Component
public class JwtUtil {
    @Value("${jwt.secret}") private String secret;
    @Value("${jwt.expiration}") private long expiration;

    public String generateToken(UserDetails user) {
        return Jwts.builder()
            .setSubject(user.getUsername())
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + expiration))
            .claim("roles", user.getAuthorities().toString())
            .signWith(Keys.hmacShaKeyFor(secret.getBytes()), SignatureAlgorithm.HS256)
            .compact();
    }

    public String extractUsername(String token) {
        return getClaims(token).getSubject();
    }

    public boolean validateToken(String token, UserDetails user) {
        return extractUsername(token).equals(user.getUsername()) 
            && !getClaims(token).getExpiration().before(new Date());
    }

    private Claims getClaims(String token) {
        return Jwts.parserBuilder().setSigningKey(secret.getBytes())
            .build().parseClaimsJws(token).getBody();
    }
}

// Filter
@Component
public class JwtFilter extends OncePerRequestFilter {
    @Autowired JwtUtil jwtUtil;
    @Autowired UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(HttpServletRequest req, 
            HttpServletResponse res, FilterChain chain) throws ... {
        String header = req.getHeader("Authorization");
        String token = null, username = null;
        if (header != null && header.startsWith("Bearer ")) {
            token = header.substring(7);
            username = jwtUtil.extractUsername(token);
        }
        if (username != null && SecurityContextHolder.getContext()
                .getAuthentication() == null) {
            UserDetails ud = userDetailsService.loadUserByUsername(username);
            if (jwtUtil.validateToken(token, ud)) {
                UsernamePasswordAuthenticationToken auth =
                    new UsernamePasswordAuthenticationToken(ud, null, ud.getAuthorities());
                auth.setDetails(new WebAuthenticationDetailsSource().buildDetails(req));
                SecurityContextHolder.getContext().setAuthentication(auth);
            }
        }
        chain.doFilter(req, res);
    }
}

  • How do you decompose a monolith into microservices? What patterns guide decomposition?
    Hard
    ▼
    Decomposition strategies:
    1. By Business Capability — map to what the business does: Order Management, Inventory, Payments, Shipping, Notifications. Most natural alignment.
    2. By Subdomain (DDD) — identify Bounded Contexts. Each BC has its own Ubiquitous Language and model. Core domain (competitive advantage) vs Supporting domain vs Generic domain.
    3. Strangler Fig Pattern — incrementally replace monolith functionality. Route traffic to new services via API Gateway, keep monolith running until fully replaced.

    Hard
    ▼

        .payload(toJson(order))
        .status("PENDING")
        .createdAt(Instant.now())
        .build();

}

  • All types of JOINs — explain with examples and when each is appropriate
    Easy
    ▼
    -- INNER JOIN — only matching rows from both tables
SELECT o.order_id, u.user_name, o.amount
FROM orders o INNER JOIN users u ON o.user_id = u.id;

-- LEFT OUTER JOIN — all rows from left, matching from right (NULL if no match)
SELECT u.user_name, COUNT(o.order_id) AS order_count
FROM users u LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.user_name;  -- includes users with 0 orders

-- RIGHT OUTER JOIN — all rows from right, matching from left
-- (usually rewritten as LEFT JOIN by swapping tables)

-- FULL OUTER JOIN — all rows from both, NULLs where no match
SELECT u.user_name, o.order_id
FROM users u FULL OUTER JOIN orders o ON u.id = o.user_id;

-- CROSS JOIN — cartesian product (every row × every row)
SELECT p1.name, p2.name FROM products p1 CROSS JOIN products p2;

-- SELF JOIN — join table with itself (org hierarchy)
SELECT e.emp_name, m.emp_name AS manager
FROM employees e LEFT JOIN employees m ON e.manager_id = m.emp_id;

  • Subquery vs JOIN vs CTE (WITH clause) — performance and readability tradeoffs
    Medium
    ▼
    -- Correlated Subquery — runs once per outer row (SLOW for large data)
SELECT emp_name FROM employees e
WHERE salary > (SELECT AVG(salary) FROM employees WHERE dept_id = e.dept_id);

-- Better as JOIN
SELECT e.emp_name FROM employees e
JOIN (SELECT dept_id, AVG(salary) avg_sal FROM employees GROUP BY dept_id) d
  ON e.dept_id = d.dept_id AND e.salary > d.avg_sal;

-- CTE (WITH clause) — readable, optimizer may materialize or inline
WITH dept_avg AS (
    SELECT dept_id, AVG(salary) avg_sal, MAX(salary) max_sal
    FROM employees GROUP BY dept_id
),
top_earners AS (
    SELECT e.*, d.avg_sal
    FROM employees e JOIN dept_avg d ON e.dept_id = d.dept_id
    WHERE e.salary > d.avg_sal * 1.5
)
SELECT emp_name, salary, avg_sal FROM top_earners ORDER BY salary DESC;

-- Recursive CTE — hierarchy traversal (org chart)
WITH RECURSIVE emp_hierarchy AS (
    SELECT emp_id, emp_name, manager_id, 1 AS level
    FROM employees WHERE manager_id IS NULL  -- top level
    UNION ALL
    SELECT e.emp_id, e.emp_name, e.manager_id, h.level + 1
    FROM employees e JOIN emp_hierarchy h ON e.manager_id = h.emp_id
)
SELECT * FROM emp_hierarchy ORDER BY level, emp_name;

Universal 6-Step Framework

  1. Clarify requirements — Functional: what it does. Non-functional: scale, latency SLA, availability (99.9% = 8.7 hrs/year downtime). Ask: read-heavy or write-heavy?
  2. Estimate scale — DAU × actions/day = QPS. 1M DAU × 10 actions = 10M req/day ÷ 86400 = ~116 QPS. Peak = 3-5× average. Storage: # records × avg size × retention period.
  3. High-level design — Client → CDN → Load Balancer → API Gateway → Services → Databases. Draw boxes and arrows, name each component.
  4. Deep dive — Go deep on the most interesting/complex part. Database schema, API design, caching strategy, consistency guarantees.
  5. Identify bottlenecks — Single points of failure, hotspots, scaling limits. Propose: replication, sharding, caching, async processing.
  6. Discuss trade-offs — SQL vs NoSQL, consistency vs availability (CAP), sync vs async, cost vs performance.

Design 1: URL Shortener (e.g. bit.ly)

  • Requirements: Shorten URL (write ~100/sec), redirect (read ~10K/sec), expiration, analytics
  • Key decision: Base62 encode a unique ID (6 chars = 62^6 = 56B URLs). Use distributed ID gen (Snowflake) not random (collision risk)
  • DB: MySQL/PostgreSQL for URL mapping (simple K-V, needs ACID for creation). Redis for hot URL cache (read-heavy, sub-ms latency)
  • Redirect flow: Browser → DNS → LB → App → Redis cache → (miss) → DB → 301/302 redirect
  • 301 vs 302: 301 (permanent, client caches) = fewer server hits. 302 (temporary) = every click hits server = better analytics accuracy
  • Analytics: Async via Kafka → Analytics service → writes to Cassandra (time-series, high write throughput)
  • Scale to 10K RPS reads: Read replicas + Redis cluster + CDN for extremely hot URLs

Design 2: E-Commerce Backend (Java + Spring Boot)

  • Services: User Service, Product Service, Order Service, Inventory Service, Payment Service, Notification Service, Search Service
  • DB choices: Orders/Users → PostgreSQL (ACID, relational). Products → MongoDB (flexible schema, categories). Search → Elasticsearch. Sessions → Redis
  • Order creation flow: POST /orders → API Gateway (auth, rate limit) → Order Service → check inventory (sync Feign) → reserve stock → call Payment Service (sync) → on success: emit OrderCreated event (Kafka) → Notification + Analytics consume async
  • Inventory: Optimistic locking to prevent overselling. Check-and-set pattern: UPDATE inventory SET stock=stock-1 WHERE product_id=? AND stock>0
  • Failure handling: Saga pattern for distributed TX. Payment fails → compensate by releasing reserved inventory
  • Caching: Product catalog → Redis (TTL 1hr, evict on update). User session → Redis. Cart → Redis (ephemeral)
  • Search: Elasticsearch for product search. Sync via CDC (Debezium) from product DB → Kafka → ES consumer

Design 3: Real-Time Notification Service

  • Requirements: Send email/SMS/push, 10M notifications/day, <5 sec delivery, at-least-once delivery, user preference management
  • Architecture: Event sources (Order, Payment, Promo) → Kafka → Notification Orchestrator → route by user preference (MongoDB) → channel-specific senders (Email via SES, SMS via Twilio, Push via FCM)
  • Deduplication: Redis SETNX with 24h TTL on notification ID — prevents duplicates on retry
  • Rate limiting: Max N notifications/user/day. Redis counter with daily expiry
  • Retry: Exponential backoff: 1s → 2s → 4s → 8s → DLQ after 5 attempts
  • Tracking: Delivery status (sent/delivered/failed/bounced) persisted to MongoDB + webhook from providers
  • Scale: Kafka partitioning by userId ensures ordered delivery per user. Horizontal scaling of senders

Core Concepts — Drop These in Any Design Discussion

  • CAP Theorem: Can't have Consistency + Availability + Partition Tolerance simultaneously. MongoDB = CP, Cassandra = AP, RDBMS = CA (no partition tolerance)
  • Consistent Hashing: Distributes data across nodes. Adding/removing node only remaps ~K/n keys. Used in Redis Cluster, Cassandra, load balancers
  • CQRS: Separate read model from write model. Write to normalized DB, project to denormalized read DB. Good for high read:write ratio
  • Event Sourcing: Store events not state. Rebuild state by replaying events. Audit trail for free. Kafka + MongoDB as event store
  • Caching strategies: Cache-aside (lazy, most common), Write-through (sync write to cache+DB), Write-behind (async), Read-through (cache proxies DB)
  • DB Sharding: Horizontal partition by shard key. Range-based (easy range queries, hot spots), Hash-based (even distribution, no range queries), Directory-based (flexible, lookup overhead)
  • Two-phase commit: Coordinator asks all nodes to prepare → if all OK → commit. Blocks on coordinator failure. Avoid — use Saga instead
  • Bloom Filter: Space-efficient probabilistic structure. "Definitely not in set" or "probably in set". Use for cache negative lookups (check Bloom before DB hit)

public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> seen = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (seen.containsKey(complement)) 
            return new int[]{seen.get(complement), i};
        seen.put(nums[i], i);
    }
    return new int[]{};
}

// Group Anagrams — LC 49
public List<List<String>> groupAnagrams(String[] strs) {
    Map<String, List<String>> map = new HashMap<>();
    for (String s : strs) {
        char[] chars = s.toCharArray();
        Arrays.sort(chars);
        String key = new String(chars); // sorted = canonical form
        map.computeIfAbsent(key, k -> new ArrayList<>()).add(s);
    }
    return new ArrayList<>(map.values());
}

// Frequency + Top K — LC 347
public int[] topKFrequent(int[] nums, int k) {
    Map<Integer, Integer> freq = new HashMap<>();
    for (int n : nums) freq.merge(n, 1, Integer::sum);
    
    // Min-heap of size k — O(n log k)
    PriorityQueue<int[]> pq = new PriorityQueue<>((a,b) -> a[1]-b[1]);
    for (var e : freq.entrySet()) {
        pq.offer(new int[]{e.getKey(), e.getValue()});
        if (pq.size() > k) pq.poll();
    }
    return pq.stream().mapToInt(a -> a[0]).toArray();
}
  • Sliding Window — longest substring, minimum window, subarray sum
    Medium
    ▼
    // Longest substring without repeating chars — LC 3 — O(n)
public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> lastSeen = new HashMap<>();
    int maxLen = 0, left = 0;
    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (lastSeen.containsKey(c)) {
            left = Math.max(left, lastSeen.get(c) + 1); // shrink window
        }
        lastSeen.put(c, right);
        maxLen = Math.max(maxLen, right - left + 1);
    }
    return maxLen;
}

// Subarray sum equals K — LC 560 — O(n)
// Use prefix sum + HashMap (not sliding window — negatives possible)
public int subarraySum(int[] nums, int k) {
    Map<Integer, Integer> prefixCount = new HashMap<>();
    prefixCount.put(0, 1); // empty prefix
    int count = 0, prefixSum = 0;
    for (int n : nums) {
        prefixSum += n;
        count += prefixCount.getOrDefault(prefixSum - k, 0);
        prefixCount.merge(prefixSum, 1, Integer::sum);
    }
    return count;
}

// Fixed window max — max sum of k-size subarray
public int maxSumSubarray(int[] nums, int k) {
    int windowSum = 0;
    for (int i = 0; i < k; i++) windowSum += nums[i];
    int maxSum = windowSum;
    for (int i = k; i < nums.length; i++) {
        windowSum += nums[i] - nums[i-k]; // slide: add new, remove old
        maxSum = Math.max(maxSum, windowSum);
    }
    return maxSum;
}
  • BFS / DFS — trees, graphs, islands problem

    // BFS — Level-order traversal + shortest path template
public List<List<Integer>> levelOrder(TreeNode root) {
    List<List<Integer>> result = new ArrayList<>();
    if (root == null) return result;
    Queue<TreeNode> q = new ArrayDeque<>();
    q.offer(root);
    while (!q.isEmpty()) {
        int size = q.size(); // freeze level size
        List<Integer> level = new ArrayList<>();
        for (int i = 0; i < size; i++) {
            TreeNode node = q.poll();
            level.add(node.val);
            if (node.left != null) q.offer(node.left);
            if (node.right != null) q.offer(node.right);
        }
        result.add(level);
    }
    return result;
}

// Number of Islands — LC 200 — BFS approach
public int numIslands(char[][] grid) {
    int islands = 0;
    int rows = grid.length, cols = grid[0].length;
    for (int r = 0; r < rows; r++) {
        for (int c = 0; c < cols; c++) {
            if (grid[r][c] == '1') {
                islands++;
                bfsSink(grid, r, c, rows, cols); // sink the island
            }
        }
    }
    return islands;
}
private void bfsSink(char[][] grid, int r, int c, int rows, int cols) {
    Queue<int[]> q = new ArrayDeque<>();
    q.offer(new int[]{r, c});
    grid[r][c] = '0';
    int[][] dirs = {{0,1},{0,-1},{1,0},{-1,0}};
    while (!q.isEmpty()) {
        int[] cur = q.poll();
        for (int[] d : dirs) {
            int nr = cur[0]+d[0], nc = cur[1]+d[1];
            if (nr>=0 && nr<rows && nc>=0 && nc<cols && grid[nr][nc]=='1') {
                grid[nr][nc] = '0';
                q.offer(new int[]{nr, nc});
            }
        }
    }
}

    // Standard binary search — O(log n)
public int binarySearch(int[] nums, int target) {
    int left = 0, right = nums.length - 1;
    while (left <= right) {
        int mid = left + (right - left) / 2; // avoids overflow
        if (nums[mid] == target) return mid;
        else if (nums[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return -1;
}

// First/Last position — find leftmost occurrence
public int firstPosition(int[] nums, int target) {
    int left = 0, right = nums.length - 1, result = -1;
    while (left <= right) {
        int mid = left + (right - left) / 2;
        if (nums[mid] == target) { result = mid; right = mid - 1; } // go left
        else if (nums[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return result;
}

// Search in rotated sorted array — LC 33
public int searchRotated(int[] nums, int target) {
    int l = 0, r = nums.length - 1;
    while (l <= r) {
        int m = l + (r - l) / 2;
        if (nums[m] == target) return m;
        if (nums[l] <= nums[m]) { // left half sorted
            if (nums[l] <= target && target < nums[m]) r = m - 1;
            else l = m + 1;
        } else { // right half sorted
            if (nums[m] < target && target <= nums[r]) l = m + 1;
            else r = m - 1;
        }
    }
    return -1;
}

class LRUCache {
    private final LinkedHashMap<Integer, Integer> cache;
    
    public LRUCache(int capacity) {
        // accessOrder=true: get() moves entry to end (most recent)
        cache = new LinkedHashMap<>(capacity, 0.75f, true) {
            @Override
            protected boolean removeEldestEntry(Map.Entry<Integer, Integer> eldest) {
                return size() > capacity;
            }
        };
    }
    public int get(int key) { return cache.getOrDefault(key, -1); }
    public void put(int key, int value) { cache.put(key, value); }
}

// LRU Cache — HashMap + Doubly Linked List (interview-preferred approach)
class LRUCacheManual {
    private final int cap;
    private final Map<Integer, Node> map = new HashMap<>();
    private final Node head = new Node(0,0), tail = new Node(0,0);
    
    public LRUCacheManual(int cap) {
        this.cap = cap;
        head.next = tail; tail.prev = head;
    }
    
    public int get(int key) {
        Node n = map.get(key);
        if (n == null) return -1;
        remove(n); addToFront(n); return n.val;
    }
    
    public void put(int key, int val) {
        if (map.containsKey(key)) remove(map.get(key));
        Node n = new Node(key, val);
        addToFront(n); map.put(key, n);
        if (map.size() > cap) {
            Node lru = tail.prev;
            remove(lru); map.remove(lru.key);
        }
    }
    
    private void remove(Node n) { n.prev.next=n.next; n.next.prev=n.prev; }
    private void addToFront(Node n) { n.next=head.next; n.prev=head; head.next.prev=n; head.next=n; }
    
    static class Node { int key, val; Node prev, next;
        Node(int k,int v){key=k;val=v;} }
}

    // Valid Parentheses — LC 20
public boolean isValid(String s) {
    Deque<Character> stack = new ArrayDeque<>(); // use Deque not Stack
    Map<Character, Character> pairs = Map.of(')', '(', ']', '[', '}', '{');
    for (char c : s.toCharArray()) {
        if (!pairs.containsKey(c)) stack.push(c);
        else if (stack.isEmpty() || stack.pop() != pairs.get(c)) return false;
    }
    return stack.isEmpty();
}

// Monotonic Stack — Next Greater Element
public int[] nextGreaterElement(int[] nums) {
    int n = nums.length;
    int[] result = new int[n];
    Arrays.fill(result, -1);
    Deque<Integer> stack = new ArrayDeque<>(); // stores indices
    for (int i = 0; i < n; i++) {
        while (!stack.isEmpty() && nums[stack.peek()] < nums[i]) {
            result[stack.pop()] = nums[i]; // found next greater
        }
        stack.push(i);
    }
    return result;
}

// Daily Temperatures — LC 739
public int[] dailyTemperatures(int[] temps) {
    int[] result = new int[temps.length];
    Deque<Integer> stack = new ArrayDeque<>(); // monotonic decreasing
    for (int i = 0; i < temps.length; i++) {
        while (!stack.isEmpty() && temps[i] > temps[stack.peek()]) {
            int prevIdx = stack.pop();
            result[prevIdx] = i - prevIdx;
        }
        stack.push(i);
    }
    return result;
}
